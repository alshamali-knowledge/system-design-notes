# The Physics of Latency: A Systems-Level Guide to Architecting Resilient Pull CDNs

## PHASE 1: THE FOUNDATION

The motivation for developing Content Delivery Networks (CDNs) stems directly from the physical limitations of the internet itself . When a user requests content from a web server located far away, the data must traverse numerous network hops, each introducing a small but cumulative delay known as latency. This delay degrades the user experience, especially for media-rich applications like video streaming or interactive websites . Furthermore, all this traffic originates from a single point, creating immense demand on the server's bandwidth and processing power, which increases operational costs and introduces a single point of failure . A CDN addresses these challenges by distributing content across a geographically dispersed network of servers, bringing the content closer to the end-user and thereby reducing latency and origin load . The core mechanism enabling this distribution is caching. Caching involves storing copies of frequently accessed resources on these distributed servers, allowing subsequent requests for that resource to be served directly from the nearby server instead of the distant origin . This paradigm shift is not merely about speed; it is also a powerful cost optimization strategy, as serving content from the edge reduces expensive data egress fees from the origin server .

To understand the specific role of a Pull CDN, it is useful to first establish an intuitive mental model. Consider a public library system. In a traditional setup, if a book is not available at your local branch, you would have to request it from the main central library. You wait for the book to be retrieved, shipped, and delivered to you. A Pull CDN operates on a similar principle. It is a system where content is fetched from the original source—the "origin server"—to the "edge" only after a user makes a request for it and the content is not already present in the local cache . The act of requesting the content "pulls" it from the origin. This contrasts with a Push CDN, where a courier proactively delivers new bestseller books to every local library branch *before* anyone requests them, ensuring they are always available for immediate checkout . This analogy highlights the fundamental difference in timing and control. A Pull CDN is reactive, waiting for demand to manifest before acting, while a Push CDN is proactive, anticipating demand.

With this foundation, we can introduce the precise definitions of the core concepts. A **Pull CDN** is a content delivery architecture where content is transferred from an origin server to edge servers only upon a direct request from a client that results in a cache miss . The primary components of this system are the **Origin Server**, the source of the original content; the **CDN Edge Nodes** (or Points of Presence), which are the distributed servers that cache content; and the **Client**, the end-user making the request . The process begins when a client sends a request for a resource, such as `https://example.com/video.mp4` . This request is routed to the nearest or most optimal CDN edge node via techniques like Anycast DNS . The edge node checks its local cache to see if it has a copy of the requested file . If the file is found, this is called a **Cache Hit**. The edge node immediately serves the cached content back to the client, completing the transaction with minimal latency because the data never needs to travel back to the origin server . This is the ideal "Happy Path" of the system. However, if the file is not in the cache, this is a **Cache Miss** . In response to a cache miss, the edge node acts as a client itself: it generates a new request to the origin server, retrieves the full resource, stores a copy in its local cache, and then forwards that resource to the original end-user client . This entire sequence—miss, fetch, cache, serve—is the fundamental workflow of a pull-based system.

This "Happy Path," characterized by a cache hit, is what defines the success of a CDN. Its effectiveness is measured by a single, crucial metric: the **Cache Hit Ratio**. This is the percentage of all incoming requests that result in a cache hit . For example, a cache hit ratio of 95% means that for every 100 requests, 95 were served from the CDN's edge cache, and only 5 required a trip back to the origin . This metric is considered the single most important number in CDN design because it directly correlates with both performance and cost savings . A high cache hit ratio translates to lower latency for the vast majority of users and significantly reduced bandwidth consumption and load on the origin infrastructure . Another closely related metric is **Origin Offload**, which measures the total volume of data served from the CDN's cache versus the total data served from the origin . It quantifies the financial benefit by showing the reduction in costly origin egress traffic . Together, these metrics provide a clear picture of whether the CDN is operating efficiently. The goal of any CDN configuration is to maximize the cache hit ratio and, consequently, the origin offload .

> 🔑 **Key Takeaway:** A Pull CDN is a reactive content delivery system that fetches content from an origin server only when a user requests it and the content is not already cached at the edge. The "Happy Path" of a cache hit provides low-latency, high-efficiency service, and its success is primarily measured by the Cache Hit Ratio.

## PHASE 2: THE REALITY

While the "Happy Path" of a cache hit is efficient and fast, the reality of operating a pull CDN is defined by its "unhappy path": the cache miss. Every cache miss triggers a cascade of events that introduce latency and place a load on the origin server, which can become a significant operational challenge . The most immediate consequence is higher latency for the first user or group of users accessing a piece of content . While serving from the edge might take milliseconds, the round-trip time (RTT) to the origin plus the time required for the origin to generate and send the response adds a substantial penalty . This initial request penalty is the primary trade-off of using a pull CDN; it prioritizes efficient delivery for repeat requests over instantaneous availability for the first one . Beyond initial latency, the more insidious problem is the unpredictable load spike placed on the origin server . Each cache miss forces the origin to handle a request that it would otherwise never have seen. If many users request the same infrequently accessed content simultaneously, the origin can be subjected to a sudden, intense burst of traffic, risking throttling or even complete failure .

Managing this dynamic behavior requires a set of standard industry patterns and a solid understanding of the tools available. The most fundamental tool is the **Time-To-Live (TTL)**. The TTL is a numerical value, typically expressed in seconds, that tells the CDN edge node how long a piece of content should be considered "fresh" and kept in the cache . Once the TTL expires, the content is considered stale, and the next request for it will result in a cache miss, triggering a revalidation fetch from the origin . By setting longer TTLs, an operator can increase the cache hit ratio and reduce the frequency of origin fetches, thus lowering origin load . However, this comes at the cost of reduced content freshness; the downside of a longer TTL is that users may receive outdated information for a longer period . This establishes a classic system design trade-off: balancing cache hit ratio and origin load against content freshness and latency . The second critical concept is the **cache key**. A cache key is the unique identifier used by the CDN to determine if a request is a hit or a miss . It is typically constructed from parts of the HTTP request, such as the URL path, query string parameters, and sometimes specific headers . If two requests have different cache keys, the CDN will treat them as two separate pieces of content, even if they point to the same underlying resource. A common mistake that leads to abysmally low cache hit ratios is having overly granular or inconsistent cache keys. For instance, if a CDN caches HTML pages by default, it can lead to a very low hit rate because every page might be treated as unique . Understanding and correctly configuring cache keys is therefore paramount to achieving high cache efficiency .

As an engineer navigates these realities, a critical decision point emerges: choosing between a pull and a push architecture. The choice is not arbitrary but is driven by the nature of the content and the application's requirements. The following table outlines the primary decision criteria:

| Criterion | Pull CDN | Push CDN |
| :--- | :--- | :--- |
| **Content Volatility** | High (Dynamic, personalized, frequently changing) | Low (Static assets, large files like videos) |
| **Freshness Requirement** | Lower (a brief delay for the first user is acceptable) | Higher (content must be available instantly) |
| **Origin Load Profile** | Bursty, unpredictable (on cache miss) | Predictable, controlled (during scheduled push/fill) |
| **Operational Complexity** | Simpler for dynamic content (no complex pipeline needed) | More complex (requires orchestration of uploads) |
| **Best Use Case** | APIs, blog posts, user-specific dashboards | Large static files (videos, images, JS/CSS bundles) |

This comparison reveals that pull and push are not opposing doctrines but rather complementary tools on a control panel, tuned for different conditions . A rule of thumb is to use a push CDN when content is large, static, and rarely updated, as this ensures maximum availability and minimum latency . Conversely, a pull CDN is suitable when content is dynamic and constantly changing, as pushing updates would be inefficient and difficult to manage . Many modern systems employ a hybrid approach, leveraging the strengths of both models . For example, large video files might be pushed to edge locations, while API responses and smaller dynamic assets are handled via pull caching . This nuanced perspective is essential for designing a robust and cost-effective global delivery system.

> 🔑 **Key Takeaway:** The operational reality of a pull CDN is governed by the trade-offs between latency, origin load, and content freshness. Mastering the configuration of TTLs and cache keys is essential for optimizing performance, and the choice between pull and push architectures depends critically on the volatility and freshness requirements of the content being delivered.

## PHASE 3: THE SCALE

At a global scale, the individual cache misses that are manageable at a small level coalesce into a systemic risk. The distributed nature of a CDN, while providing resilience and low latency, also creates new failure modes that are not apparent in simpler systems. The most prominent of these is the **Cache Stampede**, also known as the **Thundering Herd Problem** . This anti-pattern occurs when a popular piece of cached content expires simultaneously across a large portion of the CDN's edge nodes . As a result, the very next wave of user requests for that content hits the CDN and, finding no valid copy in the cache, all trigger cache misses concurrently . These thousands or even millions of simultaneous requests are then sent to the origin server, often overwhelming its capacity and causing a dramatic spike in latency or even a complete outage . This phenomenon is particularly dangerous because the load spike is not caused by external attacks but by the internal mechanics of the caching system itself, making it difficult to distinguish from legitimate traffic spikes. It represents a catastrophic failure of a naive pull-caching strategy, turning the intended benefit of load distribution into a synchronized storm of requests aimed squarely at the weakest link in the chain: the origin server .

Recognizing the Cache Stampede as a critical failure mode necessitates a corresponding set of mitigation strategies. These are not just best practices but essential engineering solutions for any production-grade pull CDN. One of the most effective techniques is **Request Coalescing** or **Collapsing** . Instead of every edge node independently sending a request to the origin, the CDN is configured so that when a cache miss occurs, it puts the downstream requests on hold. Only a single request is allowed to proceed to the origin server to fetch the content. All other waiting requests are queued locally at the edge. Once the origin responds and the content is successfully cached, the single fetched copy is used to fulfill all the waiting requests, which are then released . This transforms a massive, parallelized attack on the origin into a single, manageable request, effectively neutralizing the stampede. Another widely used technique is to apply **Jittered TTLs** . Instead of setting all cache entries for a given asset to expire at the exact same moment, a small random offset (the "jitter") is added to each entry's expiration time. This causes the cache to expire in a staggered fashion over a period of time, spreading the potential load across a wider window and preventing the sharp, synchronized peak that characterizes a stampede . A more advanced variant is **Probabilistic Early Expiration**, where a small fraction of cache entries for a given object are expired slightly before their TTL, further smoothing out the renewal process .

To illustrate the real-world consequences of these dynamics, consider the following post-mortem analysis based on publicly available information regarding Cloudflare incidents in 2025 . While not a direct cache stampede, these events highlight the fragility of the systems supporting a massive pull CDN.

> 💥 **Post-Mortem: Cloudflare Outage Analysis (November 18, 2025)**
> On November 18, 2025, Cloudflare experienced a widespread service disruption affecting a significant portion of its customer base . The incident was triggered by a bug in the logic responsible for generating configurations for its Bot Management feature within the control plane . This seemingly minor issue led to a cascade of misbehavior, causing errors in the routing and processing of traffic at the edge. The control plane, which manages the state and configuration of the global network of edge servers, became unstable. Consequently, many edge servers were unable to perform their functions correctly, leading to a degradation in performance and an increase in error rates for services relying on Cloudflare's infrastructure . The incident lasted for approximately 25 minutes, during which time the reliability of the CDN service was severely compromised . The immediate fix involved reverting the configuration change to a previously stable version, restoring normal operations . The key takeaway from this event is that the stability of a pull CDN is critically dependent on the resilience of its control plane. A failure in the centralized system managing the distributed edge cache can render the entire data plane ineffective, demonstrating that the software and management layers are as vital as the caching logic itself.

Beyond the Cache Stampede, another common anti-pattern at scale is the **Over-reliance on Default Caching Logic**. Many organizations deploy a CDN without deeply understanding its inner workings, particularly around cache key construction . This can lead to artificially low cache hit ratios, resulting in excessive origin load and unnecessary costs . For instance, a developer might assume that a CDN automatically caches all static assets, only to discover that HTML pages are not cached by default, leading to every page view hitting the origin . The remedy for this is **Proactive Cache Warming**. Instead of passively waiting for users to request content and trigger cache misses, an organization can implement scripts or automated processes that periodically fetch likely-to-be-requested content and populate the CDN cache ahead of time . This is especially effective for predictable traffic patterns, such as fetching all published articles of a news site at the start of the day, ensuring that readers encounter cache hits immediately . This shifts the model from purely reactive pull to a more intelligent, hybrid approach that mitigates the risks of both low hit ratios and cache stampedes.

> ⚠️ **Anti-Pattern: Over-reliance on Default Caching Logic**
> Many organizations use a CDN without customizing its caching policies, leading to suboptimal cache hit ratios and uncontrolled origin load.
> **Consequence:** High operational costs due to poor origin offload and potential performance degradation from unexpected traffic spikes. Misunderstanding cache keys is a frequent root cause .

> 🔑 **Key Takeaway:** At scale, the aggregate behavior of cache misses becomes a systemic risk. The Cache Stampede is a critical failure mode of naive pull-caching strategies, which must be mitigated with techniques like request coalescing and jittered TTLs. Real-world failures demonstrate that the control plane supporting the CDN is as critical as the caching logic itself.

## PHASE 4: THE PHYSICS

To move beyond descriptive engineering and achieve a deep, first-principles understanding of pull CDNs, we must ground their behavior in the immutable laws of physics and mathematics. This perspective explains *why* certain phenomena, like latency growth under load, occur and allows for more precise system design and capacity planning. The performance of a pull CDN is fundamentally constrained by two domains: the network layer and the storage hardware layer. At the network layer, the Transmission Control Protocol (TCP) governs how data flows from the origin to the edge node after a cache miss. TCP includes a congestion control mechanism called **Slow Start** . When a new connection is established, TCP begins with a small transmission window and exponentially increases the amount of data it sends until it detects packet loss or reaches a predefined limit . This algorithm is designed to be conservative, probing the network for available bandwidth without overwhelming it. The rate at which TCP can fill this window is limited by the **Round-Trip Time (RTT)** of the network path; it can only send new data once an acknowledgment for the previous batch has been received .

This leads us to the **Bandwidth-Delay Product (BDP)**, a critical concept for understanding network throughput. The BDP is calculated as the product of the network link's capacity (bandwidth) and its RTT . It represents the maximum amount of data that can be "in flight" on the network path at any given moment . To achieve maximum throughput, the TCP sender's congestion window (`cwnd`) must be at least as large as the BDP . If the `cwnd` is too small, the sender will finish transmitting its data before receiving acknowledgments, leaving the high-speed pipe idle. If the `cwnd` is too large, it can overwhelm routers and switches, causing packet loss and triggering TCP's congestion avoidance mechanisms, which drastically reduce the sending rate. Therefore, the initial latency penalty of a pull CDN is not just a simple sum of RTTs; it is compounded by the time required for the TCP slow start algorithm to ramp up the sending rate to a level that can fully utilize the network path, a process dictated by the BDP of the connection .

The second domain of physical constraint is queuing theory, which provides the mathematical language to analyze waiting lines—a perfect model for any system with finite processing capacity and variable arrival rates, such as an origin server handling CDN requests . The most fundamental theorem in this field is **Little's Law**, which states that the average number of items in a system ($L$) is equal to the average arrival rate ($λ$) multiplied by the average time an item spends in the system ($W$): $L = λW$ . While powerful, a more intuitive insight from queuing theory is the relationship between utilization and latency. For a simple M/M/1 queue (one server, Poisson arrivals, exponential service times), latency remains relatively flat at low utilization levels. However, as utilization approaches 100%, the probability of the server being busy increases, causing incoming requests to spend more time waiting in the queue. This results in a sharp, non-linear increase in average waiting time, often described as the "knee of the curve" . A commonly cited rule of thumb is that when utilization exceeds **>70%**, latency begins to grow exponentially . This formulaic understanding explains why a modest increase in traffic, pushing an origin server past its optimal utilization point, can lead to a dramatic and unacceptable degradation in perceived performance. It mathematically validates the need to mitigate cache stampedes, as a sudden influx of thousands of requests can easily drive an origin server from a healthy 50% utilization to a crippling 99%.

Finally, we must consider the limits of the storage hardware itself. After the origin server prepares the data, it must be transmitted to the CDN edge node and stored in its local cache. Modern CDNs increasingly use Solid-State Drives (SSDs) for this purpose, offering much faster access than traditional hard disk drives. However, SSD performance is highly dependent on the access pattern. **Sequential workloads**, where data is read or written in a contiguous block, consistently achieve higher throughput and significantly lower latency compared to **random workloads**, where accesses are scattered across the drive . Random reads can serialize operations within the SSD's controller, leading to higher tail latencies and variability . Since a CDN cache stores fragmented objects, its workload is a mix of sequential and random I/O, and understanding this balance is key to selecting appropriate storage hardware. While not directly part of the network RTT calculation, the speed of the CPU-cache interaction, measured in nanoseconds, underscores the vast performance hierarchy within a computing system, from network delays to memory access to storage I/O . A CDN engineer must be aware of these multi-layered constraints to build a system that performs predictably under stress.

> 📊 **Formula: Kingman's Approximation**
> An approximation for the expected waiting time in a queue, highlighting the sensitivity to utilization and service time variability:
> $$W_q \approx \left( \frac{\rho^{(\sqrt{2(c_v^2+1)}-1)}}{1-\rho} \right) \times \frac{E[S]}{2}$$
> Where $\rho$ is utilization, $c_v$ is the coefficient of variation of service time, and $E[S]$ is the expected service time. This formula shows that as utilization ($\rho$) approaches 1, the waiting time ($W_q$) grows towards infinity .

> 📊 **Formula: Bandwidth-Delay Product (BDP)**
> The maximum amount of data that can be sent on a network path before receiving an acknowledgment, determining the ideal TCP congestion window size.
> $$BDP = \text{Link Bandwidth (bps)} \times \text{RTT (seconds)}$$
> The minimum TCP receive window size in bytes is the BDP divided by 8 .

> 🔑 **Key Takeaway:** The performance of a pull CDN is not abstract; it is bounded by the laws of physics and mathematics. Network constraints like TCP slow start and BDP dictate the rate of data transfer after a cache miss, while queuing theory explains the non-linear relationship between origin server utilization and latency. Storage hardware characteristics, particularly SSD read patterns, also impose physical limits on caching efficiency.

## PHASE 5: SYNTHESIS & APPLICATION

After building knowledge from foundational concepts through to first principles, the final phase is to synthesize this understanding into a practical framework for system design. The choice between a pull and push CDN architecture is not a matter of which is universally superior, but rather a deliberate trade-off based on the specific requirements of the workload. A pull CDN offers simplicity and is ideal for dynamic content where the first-touch latency is not critical. A push CDN provides guaranteed low latency for static content but requires a more complex operational pipeline. Most sophisticated global applications leverage a hybrid model, combining the best of both worlds to optimize for performance, cost, and reliability . The key to making the right architectural decisions lies in a structured evaluation of the system's needs against the inherent characteristics of each model.

The following decision matrix provides a concise framework for comparing pull and push CDNs across several critical dimensions. This matrix serves as a quick reference guide for engineers and architects when evaluating their content delivery strategy.

| Criterion | Pull CDN | Push CDN | Hybrid Approach |
| :--- | :--- | :--- | :--- |
| **Content Type** | Dynamic, personalized, frequently changing  | Static, large files (video, images, JS/CSS)  | Combines both; pushes static assets, pulls dynamic content  |
| **Freshness Requirement** | Lower (acceptable delay for first user)  | Higher (must be available instantly)  | Balances both; uses shorter TTLs for pulled content |
| **Origin Load Profile** | Bursty, unpredictable (triggered by cache misses)  | Predictable, controlled (scheduled push/fill events)  | Mitigates worst-case loads; origins are hit predictably for pushes and intermittently for pulls |
| **Cost Model** | Pay-per-use (costs associated with origin fetches and egress)  | Upfront cost (storage and bandwidth for pre-positioning content)  | Optimizes total cost by minimizing expensive origin egress for static assets |
| **Complexity** | Simpler to configure for dynamic content  | Higher complexity (requires CI/CD pipelines for content upload)  | Highest complexity (manages two distinct content delivery mechanisms) |
| **Best Use Case** | APIs, user dashboards, CMS-generated pages  | Video-on-demand, image libraries, downloadable software  | Modern web applications, SaaS platforms, and media services  |

This matrix clarifies that the choice is not binary. For example, Netflix's Open Connect network, a highly successful proprietary CDN, employs a push-like methodology for delivering its massive video catalog to edge appliances in ISP networks, ensuring low-latency playback . However, it still relies on pull mechanisms for dynamic metadata and personalized recommendations, demonstrating a mature hybrid implementation . This integrated approach allows Netflix to optimize its entire delivery stack, maximizing caching efficiency and minimizing network hops for its core content while retaining the flexibility of on-demand retrieval for other data .

To internalize this knowledge and apply it effectively, a system designer should cultivate a set of mental checkpoints. These questions are designed to probe the assumptions behind a CDN design and uncover potential failure modes before they materialize in production. They represent a checklist for thinking like an expert.

-   **What is the acceptable latency for my content's first delivery?** This question forces a direct confrontation with the primary trade-off of a pull model. If the application cannot tolerate a delay of hundreds of milliseconds or even seconds for the first user, a pure pull strategy is likely unsuitable, and a push or hybrid model is necessary .
-   **How does my cache miss pattern affect my origin's stability?** This moves beyond theoretical load and asks for a quantitative assessment. Can the origin infrastructure—servers, databases, APIs—handle a potential surge of concurrent requests? If the answer is no, the system is vulnerable to a cache stampede, and mitigation strategies like request collapsing or jittered TTLs are not optional but mandatory .
-   **Am I leveraging a hybrid approach to optimize for both cost and performance?** This encourages looking beyond a single solution. Are there opportunities to push large, static assets to the edge proactively to guarantee low latency and reduce origin load, while using a well-managed pull strategy for dynamic content? This holistic view often yields the most resilient and cost-effective architecture .
-   **How am I validating my assumptions with real-world metrics?** This checkpoint grounds the design in empirical evidence. Have you deployed the CDN to a significant portion of traffic (e.g., 20%) to measure actual performance at scale? Are your observed Cache Hit Ratios, Origin Offload percentages, and regional latencies meeting your Service Level Objectives (SLOs)? .

By systematically working through these phases—from the intuitive "library book" analogy to the rigorous mathematics of queuing theory and network physics—an engineer can develop a comprehensive, multi-layered understanding of pull CDNs. This journey equips them not only to implement caching correctly but also to anticipate its failure modes, design resilient systems that account for the underlying physics of the internet, and make informed architectural decisions tailored to the specific demands of their application.

> 🔑 **Key Takeaway:** There is no single "best" CDN model. The optimal choice is a conscious trade-off between pull and push strategies, guided by workload characteristics. A truly expert architect uses a decision matrix to evaluate options and applies a set of mental checkpoints to ensure the chosen design is both performant and resilient.
